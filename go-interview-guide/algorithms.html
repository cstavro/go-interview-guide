Algorithms

Practical algorithmic thinking for production systems. No leetcode gotchas — just the kind of problems you solve when building real backends, pipelines, and data structures.

1. Implement a sliding window rate limiter in memory.

Senior

Given a stream of requests (each with a user ID and timestamp), implement a function that checks if a request is allowed based on a sliding window limit: no more than N requests per user in the last M seconds. Optimize for memory and time.

Boilerplate 
Loading...

Test Data

rl := NewSlidingWindowRateLimiter(60*time.Second, 3)

user := "alice"
rl.Allow(user, time.Unix(0, 0))   // true
rl.Allow(user, time.Unix(10, 0))  // true
rl.Allow(user, time.Unix(20, 0))  // true
rl.Allow(user, time.Unix(30, 0))  // false (limit exceeded)
rl.Allow(user, time.Unix(70, 0)) // true (window slid, old request expired)

Hint

Store a ring buffer or circular slice of timestamps per user. On each request, drop timestamps older than now - window, then check the count. A ring buffer gives O(1) amortized time and bounded memory.

Follow-up

  • How would you evict stale users to prevent memory growth?
  • What if you need to support millions of users? How would you shard this?
  • Compare fixed window vs. sliding window. Why is sliding window "fairer"?

Context

Sliding window rate limiting is a standard backend requirement. The naive approach (storing all requests) has unbounded memory. The ring buffer approach is the production answer. The sharding follow-up is a classic distributed systems question that separates senior from staff.

Further Reading

  • pkg.go.dev: container/ring
  • pkg.go.dev: time

2. Find the top K frequent elements in a stream.

Senior

Given a stream of strings (e.g., HTTP request paths), implement a data structure that can return the top K most frequent elements at any point. The stream is too large to store entirely. Optimize for memory and query speed.

Boilerplate 
Loading...

Hint

A min-heap of size K is the classic answer. But with a stream, you also need a frequency map. Maintain a map of all seen items and a min-heap for the top K. On each observe, update the map and possibly the heap.

Follow-up

  • What if the stream is unbounded and you need to forget old data? Implement sliding window top-K.
  • How would you parallelize this across multiple goroutines?
  • Discuss the Count-Min Sketch as an approximate alternative. When is it appropriate?

Context

Top-K is a fundamental analytics problem. In production, you'll see it in request path analysis, error tracking, and APM dashboards. The exact min-heap approach works for moderate cardinalities. The Count-Min Sketch follow-up is a great conversation starter about approximate algorithms and the memory/accuracy trade-off.

Further Reading

  • pkg.go.dev: container/heap
  • pkg.go.dev: sort

3. Implement a consistent hash ring for load balancing.

Staff

Build a consistent hash ring for distributing keys across a set of nodes. Support adding and removing nodes with minimal key movement. Include virtual nodes to improve distribution uniformity. Provide a function to get the node for a given key.

Boilerplate 
Loading...

Test Scenario

ring := New([]string{"node-a", "node-b", "node-c"}, 150)
node := ring.GetNode("user:12345")
// should consistently return the same node for the same key

ring.RemoveNode("node-b")
// only keys mapped to node-b should move

Hint

Hash each virtual node (e.g., node#0, node#1) to a position on a circle. Store sorted positions in a slice. To find the node for a key, hash the key and binary search for the next position on the ring. Wrap around to the first node if needed.

Follow-up

  • How would you implement replication (a key maps to N nodes)?
  • What hash function would you use? Why not MD5 or SHA-1?
  • How does this compare to rendezvous hashing? When is each better?

Context

Consistent hashing is the foundation of distributed caching (e.g., groupcache, Redis Cluster) and distributed storage. Understanding why it works (minimal rebalancing) and the virtual node optimization (uniformity) is a staff-level expectation. The comparison with rendezvous hashing is a great interview conversation.

Further Reading

  • pkg.go.dev: hash
  • pkg.go.dev: crypto/sha256
  • pkg.go.dev: sort

4. Implement a simple interval tree for overlapping queries.

Senior

You have a set of time intervals representing scheduled maintenance windows. Implement a data structure that can efficiently find all intervals that overlap with a given query interval. Support insertion and deletion.

Boilerplate 
Loading...

Test Data

t := NewTree()
t.Insert(Interval{"a", time(1), time(5)})
t.Insert(Interval{"b", time(3), time(7)})
t.Insert(Interval{"c", time(8), time(10)})

Overlapping(Interval{"q", time(2), time(4)})
// expected: a, b

Hint

A balanced BST (e.g., augmented red-black or AVL tree) keyed by start time, where each node stores the maximum end time in its subtree. Query by checking if the left subtree could contain overlaps (max end >= query start). If not, skip the entire subtree.

Follow-up

  • What's the time complexity of insertion, deletion, and query?
  • How would you implement this if the interval count is small (e.g., < 100)?
  • What if intervals are half-open [start, end)? How does that affect overlap?

Context

Interval trees are common in scheduling, resource allocation, and collision detection. For small sets, a simple slice scan is often faster in practice due to cache locality. The augmented BST is the theoretically correct answer, but the "small set" follow-up shows practical engineering judgment.

Further Reading

  • pkg.go.dev: container/list
  • pkg.go.dev: sort

5. Find the longest common prefix among a set of strings.

Senior

Given a slice of strings, find the longest common prefix. This is useful for path matching, auto-complete, and routing. Optimize for the case where most strings share a long prefix. Handle edge cases (empty slice, empty strings).

Boilerplate 
Loading...

Test Cases

[]string{"flower", "flow", "flight"}     // "fl"
[]string{"dog", "racecar", "car"}     // ""
[]string{}                            // ""
[]string{"alone"}                     // "alone"
[]string{"", "b"}                     // ""
[]string{"prefix", "prefix", "prefix"} // "prefix"

Hint

Sort the strings first. Then only compare the first and last strings — the longest common prefix of the whole set must be the common prefix of these two extremes. This is O(N log N) for sorting but only O(M) for the comparison, where M is the length of the shortest string.

Follow-up

  • What's the time and space complexity of your solution?
  • How would you solve this if the strings are in a Trie and you can't read them all into memory?
  • What if the strings are UTF-8? Does your solution handle multi-byte characters correctly?

Context

This appears in routing (HTTP path prefixes), logging (finding common log prefixes), and configuration management. The sort-then-compare approach is a neat optimization that many candidates miss. The Trie follow-up is a great segue into data structure design.

Further Reading

  • pkg.go.dev: sort
  • pkg.go.dev: strings
  • pkg.go.dev: unicode/utf8

6. Implement a MergeIterator over sorted iterators.

Staff

You have N sorted iterators (e.g., from database shards or log files). Implement a merge iterator that yields all elements in sorted order. The iterator should be lazy — it shouldn't materialize the entire result set. Each source iterator provides Next() and Peek() methods.

Boilerplate 
Loading...

Test Scenario

iter1: [1, 3, 5]
iter2: [2, 4, 6]
iter3: [0, 7]

merged: [0, 1, 2, 3, 4, 5, 6, 7]

Hint

A min-heap of size N is the natural structure. Each heap element stores the current value and a reference to the source iterator. On Next(), pop the smallest, advance its iterator, and push the new value back.

Follow-up

  • What if the iterators are not sorted? How would you detect this?
  • How would you handle duplicate removal across sources?
  • What if the sources are remote (network latency)? How would you prefetch?

Context

Merge iteration is the core of database query execution, log aggregation, and map-reduce shuffles. The lazy requirement is crucial for production — you can't materialize a billion rows. The network prefetch follow-up is a real distributed systems concern.

Further Reading

  • pkg.go.dev: container/heap
  • pkg.go.dev: io

7. Implement an LRU cache with O(1) operations.

Senior

Build an in-memory LRU cache supporting Get, Set, and Delete with O(1) average time complexity. It must be safe for concurrent access. Include a max size and eviction policy (LRU).

Boilerplate 
Loading...

Hint

Combine a hash map (for O(1) lookup) with a doubly linked list (for O(1) LRU tracking). The standard library's container/list works well. Protect with a sync.RWMutex or sync.Mutex.

Follow-up

  • How would you add a TTL (time-to-live) per entry?
  • What if you want to support approximate LRU (e.g., for very high throughput)?
  • How would you persist this cache to disk on shutdown and restore on startup?

Context

LRU is the most common caching interview question for a reason — it's a realistic combination of data structures and concurrency. The TTL follow-up is a real production requirement. The approximate LRU follow-up references Redis's volatile-lru and allkeys-lru eviction policies.

Further Reading

  • pkg.go.dev: sync
  • Redis: Eviction Policies

8. Implement a diff algorithm for two sorted slices.

Staff

Given two sorted slices of strings (e.g., old and new lists of user IDs), compute the difference: elements only in A, elements only in B, and elements in both. Optimize for minimal allocations and single-pass where possible.

Boilerplate 
Loading...

Test Data

a := []string{"a", "c", "d", "f"}
b := []string{"b", "c", "e", "f"}

// Expected:
// Added:   ["b", "e"]
// Removed: ["a", "d"]
// Common:  ["c", "f"]

Hint

Two-pointer technique. Walk both slices simultaneously. If a[i] < b[j], a[i] is removed. If a[i] > b[j], b[j] is added. If equal, it's common. This is O(N+M) time and O(1) extra space (excluding output).

Follow-up

  • What if the slices are not sorted? What's the best approach then?
  • How would you handle this if the slices are too large to fit in memory?
  • What if you need to stream the results instead of materializing them?

Context

Diff is a core operation in reconciliation loops (e.g., Kubernetes controllers), database sync, and cache invalidation. The two-pointer approach is elegant and efficient. The streaming follow-up is a real concern for large datasets and shows systems thinking.

Further Reading

  • pkg.go.dev: slices (Go 1.21+)
  • pkg.go.dev: sort

9. Binary search in a sorted array.

Junior

Given a sorted array of integers and a target value, return the index of the target. If the target is not present, return -1.

Boilerplate 
Loading...

Test Data

nums := []int{-1, 0, 3, 5, 9, 12}
Search(nums, 9) // 4
Search(nums, 2) // -1

Hint

Maintain left and right pointers and repeatedly cut the search space in half. Watch for infinite loops when updating pointers.

Follow-up

  • What is the time and space complexity?
  • How would you find the first or last occurrence of a duplicate target?
  • Can you implement the lower-bound variant?

Context

Binary search is far more versatile than finding elements. It underpins lower/upper bound queries, binary search on answers, and many optimization problems.

Further Reading

  • pkg.go.dev: sort

10. Merge sort.

Intermediate

Implement merge sort: given a slice of integers, return a new sorted slice. Do not mutate the input slice.

Boilerplate 
Loading...

Test Data

[]int{5, 2, 3, 1, 4} // [1, 2, 3, 4, 5]

Hint

Recursively split the slice in half, sort each half, and merge the sorted halves. The merge step uses two pointers.

Follow-up

  • What is the time and space complexity?
  • How would you make it in-place?
  • When is merge sort preferable to quicksort?

Context

Merge sort is stable and predictable, making it useful for external sorting and linked lists. It also teaches the divide-and-conquer pattern.

11. Coin change.

Intermediate

You are given coins of different denominations and a total amount. Return the fewest number of coins needed to make up that amount. If impossible, return -1.

Boilerplate 
Loading...

Test Data

coins = [1, 2, 5], amount = 11 // 3 (5 + 5 + 1)
coins = [2], amount = 3          // -1

Hint

Use dynamic programming. dp[i] = minimum coins to make amount i. Transition: dp[i] = min(dp[i - coin] + 1) for each coin.

Follow-up

  • What is the time and space complexity?
  • How would you reconstruct the actual coins used?
  • Can you optimize the space?

Context

Dynamic programming is a core algorithmic technique. Coin change is a classic unbounded knapsack problem and a great way to discuss memoization and tabulation.

Further Reading

  • pkg.go.dev: math

12. N-Queens.

Senior

The n-queens puzzle is the problem of placing n queens on an n x n chessboard such that no two queens attack each other. Return all distinct solutions.

Boilerplate 
Loading...

Test Data

n = 4 // 2 solutions
n = 1 // 1 solution

Hint

Use backtracking. Place one queen per row, and track occupied columns and diagonals. When you reach the last row, record the board.

Follow-up

  • What is the time complexity?
  • How would you count solutions without generating all boards?
  • Can bit manipulation speed it up?

Context

Backtracking problems test your ability to explore a state space methodically and prune invalid branches early. N-Queens is the canonical example.
