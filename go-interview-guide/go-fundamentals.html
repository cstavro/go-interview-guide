Go Fundamentals

Core language mechanics, memory, and idioms. The expectation at senior and staff levels is not just correct code, but the ability to explain trade-offs and edge cases.

1. Implement a function that safely appends to a slice while avoiding accidental sharing of the underlying array.

Senior

Write a function AppendUnique(dest, src []int) []int that appends elements from src to dest without causing the caller to accidentally share the backing array with other slices. Include a demonstration that proves the original dest is unaffected if the returned slice grows beyond dest's capacity.

Expected Behavior

original := []int{1, 2, 3}
result := AppendUnique(original, []int{4, 5, 6, 7, 8})
// result should be [1 2 3 4 5 6 7 8]
// original should remain [1 2 3] regardless of capacity

Hint

Think about append's behavior when it exceeds capacity. What happens to the backing array? How can you force a new allocation?

Follow-up

  • What if dest has enough capacity? Should you still copy?
  • How does this relate to the "slice header" (pointer, len, cap)?
  • Would you implement this differently for a library vs. application code?

Context

Slice aliasing is a common source of subtle bugs in Go. At senior levels, you should be able to reason about memory sharing and when to force copies. This question tests whether you understand the underlying array mechanics, not just the syntax.

Hint

5. Implement a generic pool that limits object reuse to the same type.

Senior

Go's sync.Pool is not type-safe and can suffer from type assertions. Build a generic, type-safe object pool Pool[T] with Get() T and Put(T) methods. It should use sync.Pool under the hood but expose a clean API. Include a benchmark comparing it to a naive allocation.

Hint

Store a sync.Pool inside your generic struct. The trick is that sync.Pool stores any, so you can type-assert on Get. Since the pool is per-type, the assertion is always safe.

Follow-up

  • When is sync.Pool actually faster than GC? What are the gotchas?
  • How does sync.Pool interact with GC cycles? Explain the clearing behavior.
  • Would you use this for a request-scoped object? Why or why not?

Context

Object pooling is a classic optimization, but Go's GC is very efficient. The real value of this question is understanding when pooling helps (high-frequency, short-lived allocations) and when it hurts (complex object lifetimes, memory retention). At senior level, you should know the trade-offs, not just the API.

Further Reading

  • pkg.go.dev: sync.Pool
  • Effective Go: Leaky Buffer Pattern
  • Go GC Guide

Hint

Apply defaults first, then loop through options. Return an error from the option function itself so validation can happen at configuration time. Consider what happens if the same option is passed twice.

Follow-up

  • How would you expose an option that sets a field conditionally (e.g., WithPort only if not already set)?
  • What are the downsides of functional options vs. a config struct?
  • How would you support option composition for sub-services?

Context

Functional options is one of Go's most idiomatic patterns for clean APIs. It was popularized by Rob Pike and is used throughout the standard library (e.g., tls.Config predates it, but many third-party libraries use it). Understanding when to use it vs. a config struct shows API design maturity.

Further Reading

  • Rob Pike: Self-referential Functions and the Design of Options
  • Effective Go: Interfaces
  • Dave Cheney: Functional Options for Friendly APIs
