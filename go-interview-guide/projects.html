Realistic project problems that involve debugging or extending a small existing application. These are designed for AI-assisted coding interviews — the problems encourage small, focused AI usage rather than simply asking an agent to "fix this" or "implement this".

1. Debug and fix a goroutine leak in an HTTP retry client.

Senior

A small HTTP client with retry logic is leaking goroutines. The client uses goroutines for concurrent requests, but when the context is cancelled, the goroutine blocks forever trying to send on a channel. The tests pass, but the benchmark shows a growing goroutine count.

Test Scenario

go test -bench=. -run=^$ -count=5

Hint

When the ctx.Done() branch is selected, the goroutine is still blocked on respCh <- resp or errCh <- err. Use a buffered channel or a select with a done channel in the goroutine.

Follow-up

  • How would you detect this leak in production? Mention runtime.NumGoroutine and pprof.
  • What is the go.uber.org/goleak library and how does it help?
  • How would you refactor this to avoid goroutines per retry entirely?

Context

Goroutine leaks are a common source of production OOMs. The classic pattern is a goroutine that sends on a channel that nobody receives. This question tests whether you can reason about channel lifetimes and context cancellation. The fix is simple once you see the pattern, but the bug is subtle enough to hide in code review.

Further Reading

  • go.uber.org/goleak
  • Go Blog: Pipelines and Cancellation

Context

Profiling is a fundamental skill for senior engineers. In production, you'll use pprof to find hot paths. The key insight is that string splitting in Go allocates a new slice for every call. 3. Fix middleware order and add timing.

Senior

Hint

Middleware wraps from the outside in. The last middleware applied is the first to execute. If you want logging to run before recovery (so it logs even when recovery happens), apply logging after recovery in the constructor.

Follow-up

  • How would you make the middleware chain configurable at runtime (e.g., via a slice)?
  • What happens if a middleware panics? How do you protect the outer layer?
  • How would you implement a middleware that adds a request ID to the context and logs it?

Context

Middleware ordering is a common source of confusion in Go. The wrapping order determines execution order. This is a realistic bug because it's easy to get wrong and hard to catch in tests unless you specifically test panic paths. The timing middleware addition is a typical "extend the codebase" task that mirrors production work.

Follow-up

  • What if the implementation used a channel instead of a mutex? How would the test change?
  • How would you detect flakiness in CI? Mention go test -count=N and stress testing.
  • What is the difference between a flaky test and a flaky implementation?

Context

Flaky tests are a productivity killer. The root cause is often a misunderstanding of concurrency primitives. This question tests whether you can identify that the test is the problem, not the implementation. The constraint "don't change the implementation" is realistic — you often can't change production code, but you can improve the test.

The Flush() addition is a realistic extension request — adding features to existing code without breaking the current behavior.

7. Debug broken cursor pagination encoding.

Staff

This REST API handler uses cursor-based pagination. The cursor encodes a timestamp and an ID. The bug is subtle: json.Marshal for an int64 timestamp is parsed as float64 by default, which loses precision for large values (anything above 2^53). This causes pagination to return wrong results or loop forever.

Buggy Code 
Loading...

Hint

JSON numbers are parsed as float64 by encoding/json. The maximum safe integer in float64 is 2^53 (≈9 quadrillion). time.Now().UnixNano() is around 1.7 quintillion, well above the limit. Use a custom binary encoding (e.g., encoding/binary or fmt.Sprintf with base64) instead of JSON.

Follow-up

  • Why does encoding/json use float64 for numbers? Could you force it to use json.Number?
  • How would you encode the cursor if it had more fields (e.g., sort order, filters)?
  • Should cursors be signed or encrypted? Discuss the security implications.

Context

Cursor pagination is the standard for high-performance APIs. The precision bug is a classic because it only manifests with large timestamps — it might work fine in development and fail in production. Understanding JSON number precision is a staff-level expectation. The security follow-up about signing cursors is a real concern for public APIs.

Further Reading

  • pkg.go.dev: encoding/json
  • pkg.go.dev: encoding/binary
  • Twitter API: Pagination
