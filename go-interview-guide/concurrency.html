Concurrency

Go's concurrency model is a core differentiator. At senior and staff levels, expect to reason about correctness, performance, and the memory model under pressure.

1. Implement a rate limiter using the token bucket algorithm.

Senior

Write a TokenBucket that allows bursts up to a maximum capacity but refills at a fixed rate. Provide Allow() and AllowN(n int) methods. Must be safe for concurrent use. Include a test that demonstrates a burst followed by throttling.

Test Scenario

bucket := NewTokenBucket(5, 1*time.Second) // capacity 5, 1 token/sec
fmt.Println(bucket.AllowN(5)) // true (burst)
fmt.Println(bucket.Allow())   // false (throttled)
// wait 1 second
fmt.Println(bucket.Allow())   // true (refilled)

Hint

Track the last time tokens were consumed and compute how many tokens should have refilled since then. Use a mutex or atomic operations. Lazy evaluation is your friend — you don't need a background goroutine.

Follow-up

  • How would you implement a "Wait" variant that blocks until tokens are available?
  • What's the difference between token bucket and leaky bucket? When to use which?
  • How would you distribute this across a cluster of services?

Context

Rate limiting is a fundamental resilience pattern. The token bucket is preferred when you want to allow bursts. The lazy evaluation approach is a common interview optimization because it avoids background goroutines. Understanding the distributed version (e.g., Redis-based) shows staff-level thinking.

2. Implement a worker pool with graceful shutdown.

Senior

Build a worker pool that processes jobs from a channel. It must support Start, Submit, and Stop (graceful). On Stop, it should finish all in-flight jobs but reject new submissions. Use a context.Context for cancellation.

Boilerplate 
Loading...

Hint

Use a jobs channel, a sync.WaitGroup for workers, and a separate channel or atomic flag to signal shutdown. The key is to close the jobs channel only after all submits are done, and to wait for workers to drain.

Follow-up

  • What happens if a job panics? How would you recover and continue?
  • How would you implement dynamic resizing (add/remove workers)?
  • Compare this to using errgroup.Group or ants library.

Context

Worker pools are a bread-and-butter concurrency pattern. The graceful shutdown requirement is what makes it interesting — it mirrors production concerns where you must drain connections before restart. The context integration is also a key Go idiom.

Further Reading

  • pkg.go.dev: golang.org/x/sync/errgroup
  • pkg.go.dev: golang.org/x/sync/semaphore
  • pkg.go.dev: context

4. Implement a timeout pattern for a function that calls multiple services.

Senior

You need to call three services (A, B, C) concurrently. The overall operation must complete within 500ms. If any service fails or times out, return an error immediately. Do not leak goroutines. Include the code to call the services and the orchestration logic.

Boilerplate 
Loading...

Hint

Use errgroup or sync.WaitGroup with a context.WithTimeout. The key is to propagate the context and to cancel remaining goroutines when the first error occurs. If you use errgroup, it handles cancellation automatically.

Follow-up

  • What if you want partial results (return whatever succeeded, plus the errors)?
  • How would you implement retry with exponential backoff per service?
  • What if the services are streamed and you need to return the first successful result?

Context

Context propagation and goroutine coordination are fundamental to Go backend services. The "no goroutine leaks" requirement is critical — in production, leaking goroutines exhausts memory and causes OOMs. This is the kind of pattern you'll use dozens of times in real systems.

Further Reading

  • pkg.go.dev: context
  • pkg.go.dev: golang.org/x/sync/errgroup
  • Go Blog: Pipelines and Cancellation

5. Explain and implement a fan-out / fan-in pattern.

Staff

Given a channel of input jobs, distribute them to N workers, collect results, and return a single channel. Ensure proper shutdown when input is exhausted. The output order does not need to match input order.

Boilerplate 
Loading...

Hint

Use a sync.WaitGroup to wait for all workers to finish, then close the results channel. The main goroutine should range over results until it's closed. Separate the input channel lifecycle from the output channel lifecycle.

Follow-up

  • How would you preserve input order in the output?
  • What if you want to limit the number of in-flight jobs (backpressure)?
  • How would you propagate errors from workers back to the main goroutine?

Context

Fan-out/fan-in is the bread and butter of pipeline parallelism. The order-preserving follow-up is a common production requirement. The error propagation follow-up is where it gets interesting — most real pipelines need to handle partial failures, and the naive channel pattern doesn't handle that well.

Further Reading

  • Go Blog: Pipelines and Cancellation
  • pkg.go.dev: sync.WaitGroup

Senior

Boilerplate

Hint

A channel with buffer size N is the natural representation. Sending to the channel is Acquire, receiving is Release. Make sure to handle the case where Release is called more than Acquire.

Follow-up

  • What happens if you forget to call Release? How would you detect this?
  • Implement AcquireContext that supports cancellation.
  • When is a semaphore better than a worker pool? When is it worse?

Context

Semaphores are a classic concurrency primitive. The channel-based implementation is a great interview question because it demonstrates that you understand channels as more than just message passing — they are synchronization primitives. The AcquireContext follow-up is a real-world requirement for any production system.

Context

This is the canonical Go concurrency puzzle, often attributed to Dmitry Vyukov. The correct answer is that there is no synchronization, so the behavior is undefined. This question separates people who know goroutines from people who understand the Go memory model. Staff engineers should be able to discuss why the memory model is designed this way and the historical evolution of atomic guarantees.

8. Implement a pub/sub system with topic filtering.

Staff

Build a pub/sub system where subscribers can subscribe to topics with wildcards (e.g., orders.* matches orders.created and orders.updated). Support concurrent publishers and subscribers. Subscribers receive messages via a channel. Ensure slow subscribers don't block fast publishers.

Boilerplate 
Loading...

Hint

Store subscribers in a map protected by a mutex. Each subscriber gets a buffered channel. Use a simple glob match (e.g., split by . and compare segments). For non-blocking sends, use a select with a default case to drop messages for slow subscribers.

Follow-up

  • How would you implement exactly-once delivery? Is it possible?
  • How would you add subscriber groups (only one subscriber in a group receives a message)?
  • What are the memory implications if subscribers never unsubscribe?

Context

Pub/sub is a foundational pattern in distributed systems. The wildcards and non-blocking requirements make this a realistic problem. The follow-ups about exactly-once delivery and subscriber groups are core concepts in systems like Kafka and NATS. Staff engineers should understand why exactly-once is fundamentally impossible in distributed systems without trade-offs.
