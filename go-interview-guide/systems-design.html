Systems Design

Distributed systems, API design, resilience, and observability. The questions here are broad and open-ended — the goal is to show how you think about trade-offs, not to arrive at a single correct answer.

1. Design a URL shortener service.

Senior

Design a service like bit.ly that takes a long URL and returns a short URL. It must redirect short URLs to the original. Consider: database schema, hashing strategy, collision handling, rate limiting, analytics, and cache strategy.

Hint

Start with the data model. A simple table: (short_key, long_url, created_at, clicks). For the hash, consider base62 encoding of a distributed ID (e.g., Snowflake). For collisions, either check-and-retry or use a deterministic hash with a counter.

Follow-up

  • How would you handle custom short URLs (e.g., bit.ly/my-brand)?
  • What if the same long URL is submitted twice? Should you return the same short URL?
  • How would you shard the database? What sharding key would you use?
  • Design for analytics: how do you track clicks without blocking the redirect?

Context

URL shorteners are the quintessential systems design question. They're simple enough to design in 45 minutes but complex enough to test deep knowledge. The key differentiator is how you handle the read-heavy workload (redirects) vs. the write-heavy workload (new URLs). Senior candidates should mention caching and database choice; staff candidates should discuss sharding and eventual consistency.

Further Reading

  • Redis: Scaling
  • pkg.go.dev: hash/crc32

2. Design a distributed rate limiter.

Staff

Your service runs across 10 nodes. You need to enforce a global rate limit: 1000 requests per minute per user across all nodes. Design the system. Consider: synchronization strategy, consistency model, failure modes, and performance.

Hint

Options include: centralized Redis (simple but has a bottleneck), local token buckets with gossip/periodic sync (approximate but fast), or a distributed consensus protocol (accurate but slow). Start with the trade-offs and pick based on requirements.

Follow-up

  • What happens if Redis is down? Should you fail open or fail closed?
  • How would you handle clock skew between nodes?
  • What if the limit is per-IP instead of per-user? How does that change the design?
  • Discuss the CAP theorem implications of your design.

Context

Distributed rate limiting is a real production problem. The centralized vs. decentralized trade-off is a classic staff-level discussion. The "fail open vs. fail closed" follow-up tests your operational thinking. At staff level, you should be able to discuss how this integrates with API gateways (e.g., Kong, Envoy) and service mesh.

Further Reading

  • Redis: Distributed Locks
  • pkg.go.dev: golang.org/x/time/rate

3. Design a job queue with exactly-once semantics.

Staff

Build a job queue system where each job must be processed exactly once. Jobs can fail and be retried. Discuss: idempotency, deduplication, at-least-once delivery, and the impossibility of exactly-once in distributed systems. Provide a design that achieves "effectively exactly-once" in practice.

Hint

The key insight is that exactly-once delivery is impossible, but exactly-once processing is achievable with idempotency. The pattern is: assign a unique ID to each job, store the result keyed by ID, and deduplicate before processing. The acknowledgment and result storage must be atomic.

Follow-up

  • How would you handle job ordering? Is FIFO guaranteed?
  • What if a job is poisoned and crashes the worker every time?
  • How would you implement delayed jobs (e.g., "process this in 5 minutes")?
  • Discuss the Two Generals Problem and why it makes exactly-once delivery impossible.

Context

Job queues are the backbone of async processing. The exactly-once discussion is a classic interview litmus test for understanding distributed systems fundamentals. The "effectively exactly-once" pattern (idempotency + deduplication) is used in payment processing, inventory management, and any system where duplication is catastrophic. Staff engineers should be able to explain why naive approaches fail and what the industry standard solutions are (e.g., Kafka's idempotent producer, SQS FIFO + deduplication).

Further Reading

  • Apache Kafka Documentation
  • AWS SQS: FIFO Queues
  • pkg.go.dev: database/sql

4. Design a real-time notification system.

Senior

Design a system that pushes notifications to millions of users in real-time. Notifications can be triggered by events (e.g., a message, a price alert). Support: WebSocket, push notifications (APNs/FCM), and email. Users can set preferences per channel. Discuss fan-out, delivery guarantees, and backpressure.

Hint

Break it into: event ingestion (Kafka), fan-out (per-user queue or shared topic), and delivery (WebSocket manager, push gateway, email worker). Use a queue per user or per user shard to avoid head-of-line blocking. For WebSocket, maintain a connection registry.

Follow-up

  • What if a user has 10M followers? How do you avoid a thundering herd?
  • How do you handle a user who is offline? Do you buffer indefinitely?
  • What if a notification channel is down? How do you retry without spamming?
  • How would you implement "read receipts" across channels?

Context

Real-time notification systems are some of the most complex backend systems. The fan-out problem (1 event → N users) is a classic scalability challenge. The offline buffering question tests your understanding of bounded vs. unbounded queues. At staff level, you should discuss how to measure and optimize delivery latency (p50, p99, p99.9).

Further Reading

  • Apache Kafka Documentation
  • Apple: User Notifications
  • Firebase Cloud Messaging

5. Design an API versioning strategy for a growing service.

Senior

Your API has 1000+ clients. You need to introduce a breaking change (e.g., changing how pagination works). Design a versioning strategy that minimizes client disruption. Discuss: URL versioning vs. header versioning, backward compatibility, deprecation lifecycle, and how to enforce sunset deadlines.

Hint

URL versioning (/v1/, /v2/) is the simplest but pollutes the URL. Header versioning (Accept: application/vnd.api.v2+json) is cleaner but harder for clients. The real answer is: always be backward compatible if possible, use versioning as a last resort, and have a clear deprecation policy with automated client communication.

Follow-up

  • How would you implement a compatibility layer that translates v1 requests to v2 internally?
  • What if a client never updates? How do you handle long-tail deprecation?
  • How would you use feature flags to roll out a new API version gradually?
  • Discuss the pros and cons of gRPC vs. REST for API versioning.

Context

API versioning is a product engineering problem as much as a technical one. The best technical solution is useless if clients don't migrate. Senior engineers should understand the technical options; staff engineers should understand how to drive cross-team coordination and deprecation programs. This is a common area where technical decisions have business impact.

Further Reading

  • gRPC: Introduction
  • pkg.go.dev: net/http

Follow-up

  • What if the failure is a timeout? How do you distinguish between slow and failing?
  • How would you integrate this with a service mesh (e.g., Istio, Linkerd)?
  • Discuss the difference between circuit breaker and bulkhead pattern.

Context

Circuit breakers are one of the four pillars of microservice resilience (along with retry, timeout, and bulkhead). Understanding the state machine and the edge cases (half-open behavior, timeout vs. error) is critical. The service mesh follow-up is a staff-level question because it shows you understand the organizational trade-off between library-based and infrastructure-based resilience.

7. Design an observability pipeline for a Go service.

Staff

Your team runs 50 Go services. You need to collect logs, metrics, and traces with minimal overhead. Design the pipeline: instrumentation, collection, storage, and querying. Discuss: structured logging, metric cardinality, trace sampling, and alerting. Include code examples for the instrumentation layer.

Hint

Think about the three pillars: logs (structured, e.g., slog), metrics (Prometheus-style counters/histograms), traces (OpenTelemetry). The key is low overhead: async batching, sampling for traces, and avoiding high-cardinality metrics (e.g., don't label by user ID). For storage, use separate backends optimized for each data type.

Follow-up

  • How do you prevent high-cardinality metrics from exploding your TSDB?
  • What's the difference between logs and traces? When do they overlap?
  • How would you implement distributed tracing across services? What headers are needed?
  • How do you balance observability overhead vs. the value of the data?

Context

Observability is not just tooling — it's a design philosophy. Staff engineers are expected to have opinions on the trade-offs between instrumentation overhead and debuggability. The high-cardinality question is a common production incident root cause. The OpenTelemetry standard is increasingly the industry default, and you should understand how it integrates with Go's runtime.

8. Design a leader election mechanism for a distributed service.

Staff

You have a service that must run a background task on exactly one node at a time. Design a leader election mechanism. Discuss: split-brain prevention, network partition handling, graceful handoff, and liveness detection. Provide a Go sketch using a distributed lock (e.g., Redis Redlock, etcd, or Consul).

Hint

The core loop: acquire lock → if acquired, start leader work and a background renew goroutine → if renew fails, cancel leader context and try to re-acquire. The lock must be tied to a unique node ID to prevent a zombie node from holding the lock after a network partition heals.

Follow-up

  • What is the Redlock algorithm? What are the criticisms of it?
  • How does etcd's leader election work? Why is it more reliable than Redis?
  • What happens if the leader task takes longer than the TTL? How do you handle it?
  • Discuss the FLP impossibility result and how it applies to leader election.

Context

Leader election is the foundation of distributed coordination. It's used in Kubernetes controllers, database failover, and batch job scheduling. The Redlock controversy (Martin Kleppmann's blog post) is a famous distributed systems discussion point. At staff level, you should understand the theoretical limits (FLP impossibility) and the practical compromises (etcd, ZooKeeper, Raft-based systems). You should also be able to discuss how Go's context and goroutines make the leader loop implementation clean and idiomatic.
